# 第 1 讲：Agent 到底解决什么企业问题
Created: 2026-05-24

## 1. 先建立一个判断标准

学 Agent 开发，最容易走偏的地方是：看到模型能调用工具，就觉得什么都可以交给 Agent。

企业不是这样用 Agent 的。

企业真正关心的不是“模型像不像人”，而是：

- 能不能减少重复劳动？
- 能不能连接已有系统？
- 能不能处理非结构化输入？
- 能不能给出可追溯依据？
- 能不能知道什么时候必须交给人？
- 能不能通过测试、审计和灰度证明它可以被信任？

所以我们先给 Agent 一个实用定义：

**企业里的 Agent，是一个能围绕目标使用模型、工具、数据和状态来完成多步骤任务，并且能被测试、观测和约束的软件系统。**

这里有几个关键词：

- **目标**：不是闲聊，而是完成业务任务。
- **工具**：能查系统、读文档、创建工单、触发审批。
- **数据**：能使用企业知识库、历史记录、业务数据。
- **状态**：能记住任务进展，而不是每轮从零开始。
- **约束**：知道权限、风险、审批和拒答边界。
- **可测试**：能用 eval、trace、安全样例证明能力。

如果一个系统只是“用户问，模型答”，它最多是 chatbot 或 LLM app，不一定是 Agent。

## 2. Chatbot、Workflow、Agent、多 Agent 的区别

先把几个词分清楚。

### Chatbot

Chatbot 的核心是对话。

典型能力：

- 回答问题
- 总结内容
- 改写文本
- 根据上下文继续交流

它可能不调用工具，也可能没有明确任务状态。很多企业早期 AI 项目停留在这里：接一个知识库，让模型回答问题。

问题是：企业流程通常不只需要回答，还需要行动。

例如员工问：

> 我的报销为什么还没到账？

一个普通 chatbot 可能只能回答报销制度。真正有价值的系统需要：

- 识别这是报销进度问题。
- 查询员工身份和报销单。
- 判断当前卡在哪个节点。
- 给出原因和下一步。
- 必要时提醒审批人或创建催办任务。

这就不只是聊天了。

### Workflow

Workflow 是预定义流程。

例如：

```text
接收问题
  ↓
分类
  ↓
检索知识库
  ↓
查询业务系统
  ↓
生成建议
  ↓
风险判断
  ↓
人工审批或自动回复
```

workflow 的优点是可控、可测试、可审计。企业场景里，大部分所谓 Agent 系统，第一版都应该从 workflow 开始。

Anthropic 在 agent 工程实践里也强调：先找最简单可行方案，只有当复杂度能明显改善结果时才增加复杂度；对明确任务，workflow 通常比完全自主 agent 更可预测。

### Agent

更狭义的 Agent，是模型能动态决定下一步做什么、调用哪个工具、如何推进任务。

它适合的问题通常有这些特征：

- 任务路径不完全固定。
- 中间需要根据结果继续判断。
- 工具调用顺序可能变化。
- 目标清楚，但步骤需要探索。
- 有反馈循环，例如测试失败后继续修复。

代码库维护 Agent 就是典型例子：

```text
读 issue
  ↓
搜索相关代码
  ↓
理解失败原因
  ↓
生成修改方案
  ↓
运行测试
  ↓
根据测试结果继续修复
  ↓
输出风险说明
```

但越自主，越需要 sandbox、权限、trace、测试和人工审查。

### 多 Agent

多 Agent 是多个 Agent 分工协作，例如：

- 一个负责检索资料。
- 一个负责判断风险。
- 一个负责生成方案。
- 一个负责审查输出。

初学阶段不要过早追求多 Agent。很多多 Agent demo 只是把一个任务拆成几个角色名，看起来热闹，但没有提升可靠性，反而增加调试难度。

企业落地的顺序应该是：

```text
单次 LLM 调用
  ↓
结构化输出
  ↓
工具调用
  ↓
显式 workflow
  ↓
单 Agent
  ↓
必要时多 Agent
```

## 3. 企业 Agent 的 5 类真实落地场景

我们这门课会围绕 5 类企业场景展开。

### 3.1 客服 / 工单 / 售后 Agent

这是目前最成熟的 Agent 落地方向之一。

企业痛点：

- 用户问题量大、重复率高。
- 用户表达不规范，不能只靠关键词。
- 答案依赖订单、账号、合同、政策、历史工单。
- 某些动作有风险，例如退款、改签、停用账号、改权限。
- 必须支持人工接管和审计。

一个可落地的客服 Agent 不是这样：

```text
用户问问题 → 模型自由回答
```

而是这样：

```text
用户问题
  ↓
识别意图和风险
  ↓
查询用户与订单
  ↓
检索政策知识库
  ↓
生成处理建议
  ↓
低风险直接回复
  ↓
高风险提交人工审批
```

这类 Agent 的核心不是“回答得像客服”，而是“能正确连接业务系统，并知道什么不能自动做”。

### 3.2 企业知识库 / Research Agent

企业知识通常散落在很多地方：

- 文档
- 知识库
- 群聊
- 邮件
- 工单系统
- CRM
- 代码库
- 数据报表

普通 RAG 问答只能解决一部分问题。企业真正需要的是：

- 找到正确来源。
- 引用具体依据。
- 区分事实、推断和未知。
- 发现资料之间的冲突。
- 遵守权限。
- 知道资料是否过期。

例如员工问：

> 企业客户的退款规则是什么？销售说可以 30 天内退款，但知识库里好像写的是 7 天。

一个好的 Research Agent 应该：

- 检索退款政策文档。
- 检索销售例外政策。
- 检查文档版本和更新时间。
- 指出冲突。
- 不确定时建议人工确认。
- 输出引用来源。

它不应该在证据不足时编一个看起来合理的答案。

### 3.3 业务流程自动化 / Back-office Agent

很多企业后台流程的输入都很混乱：

- 邮件里的订单
- 图片里的表单
- PDF 合同
- 语音记录
- 手写备注
- 聊天消息

Choco 的食品分销案例就属于这一类：订单来自多种非结构化输入，Agent 的任务是把它们转成 ERP 可以处理的结构化订单，并配合置信度阈值、人工审核、监控和 eval。

这类 Agent 的关键不是“能抽字段”，而是：

- 能理解业务上下文。
- 能匹配客户、商品、SKU、历史偏好。
- 能处理缺失信息。
- 能发现异常。
- 能在低置信度时进入人工审核。

### 3.4 代码库维护 / DevEx Agent

这类 Agent 很适合工程师做进阶作品集。

企业痛点：

- 大型代码库上下文复杂。
- 新人 onboarding 慢。
- 测试补齐、代码审查、迁移重构成本高。
- on-call 排障需要跨系统查信息。

可落地方向：

- 读 issue，定位相关文件。
- 解释某段代码为什么这样写。
- 生成修改方案。
- 补测试。
- 运行测试并根据失败结果迭代。
- 输出风险和回滚建议。

但它不适合作为我们第一版课程主线。原因是代码 Agent 很容易把课程带进 IDE、patch、sandbox、测试运行和代码安全细节。我们会把它放在最后，作为“进阶作品集方向”。

### 3.5 企业 Agent 平台与治理

当一个企业里多个团队都开始做 Agent，会出现新问题：

- 谁创建了哪些 Agent？
- 每个 Agent 能访问什么数据？
- 每个 Agent 能调用什么工具？
- 工具调用有没有审计？
- prompt 和模型版本有没有管理？
- 出问题时能不能追踪和停用？
- 第三方 MCP server 是否可信？

这时 Agent 不再只是一个功能，而是需要平台治理。

治理能力包括：

- agent registry
- tool registry
- permission model
- trace and audit
- eval pipeline
- deployment gate
- incident response

这也是为什么企业级 Agent 课程必须讲测试、安全、权限和观测。

## 4. 为什么我们的主线项目选“企业内部工单与知识库 Agent”

最终项目确定为：

**Enterprise Support Workflow Agent：企业内部工单与知识库 Agent**

它不是最炫的方向，但最适合作为企业 Agent 入门主线。

原因有 5 个。

### 4.1 它足够真实

几乎每个企业都有类似问题：

- 员工不知道制度在哪里。
- 客户问题需要查历史记录。
- 一线同事需要问后台团队。
- 审批、工单、知识库、IM 分散。
- 重复问题消耗大量人力。

这个场景不是为了展示 AI，而是有明确业务成本。

### 4.2 它覆盖核心技术面

它天然需要：

- 意图识别
- 结构化输出
- RAG
- tool calling
- workflow
- state
- human-in-the-loop
- eval
- trace
- security
- production readiness

学完它，很多企业 Agent 场景都能迁移。

### 4.3 它适合理论先行

我们第一版课程先不写代码模板。工单与知识库 Agent 很适合先讲清楚：

- 业务边界
- 流程设计
- 工具设计
- 权限设计
- 测试设计
- 上线策略

后续你真正做项目时，再把这些设计落成代码。

### 4.4 它可以自然结合飞书

在中文企业环境里，飞书这类协作平台很适合作为案例参照。

飞书开放平台官方文档覆盖消息与群组、文档、知识库、多维表格、审批、服务台、任务等能力。这些能力正好可以映射到一个企业 Agent 系统：

| 企业能力 | 在 Agent 系统里的含义 |
|---|---|
| 群聊 / 单聊 | 用户入口 |
| 机器人 | Agent 对话入口或通知入口 |
| 知识库 / 文档 | RAG 数据源 |
| 多维表格 | 轻量业务数据库或工单表 |
| 审批 | human-in-the-loop |
| 服务台 | 工单系统 |
| 任务 | 后续跟进动作 |

注意：第一版课程不写飞书 API 代码。我们只是用它帮助你理解企业 Agent 会连接哪些真实系统。

### 4.5 它能扩展到代码库维护 Agent

等主线项目学完，再做代码库维护 Agent 会更稳。

因为你已经掌握了：

- 如何拆 workflow。
- 如何设计工具边界。
- 如何做 eval。
- 如何看 trace。
- 如何处理高风险动作。

代码 Agent 只是把工具从“工单、知识库、审批”换成“代码搜索、文件读取、测试运行、patch 生成”。

## 5. 主线项目的业务设定

我们先给最终项目一个清晰设定。

假设有一家 B2B SaaS 公司，内部使用飞书做协作。公司有：

- 飞书群聊：员工和客服提交问题。
- 飞书知识库：产品文档、退款政策、权限制度、SOP。
- 飞书多维表格：模拟客户、订单、工单、历史案例。
- 飞书审批：高风险动作审批。
- 飞书服务台：承载正式工单流程。

Agent 的任务是：

```text
接收问题
  ↓
识别问题类型
  ↓
判断风险等级
  ↓
检索知识库
  ↓
查询模拟业务数据
  ↓
生成处理建议和依据
  ↓
低风险：回复或创建普通工单
  ↓
高风险：提交审批或转人工
  ↓
记录 trace、成本、延迟、失败原因
```

示例问题：

```text
客户 A 说合同里承诺可以 30 天无理由退款，但我们知识库里写的是 7 天。这个单子应该怎么处理？
```

Agent 不应该直接退款。它应该：

- 判断这是高风险售后问题。
- 检索退款政策。
- 查询客户合同或历史记录。
- 找出政策冲突。
- 生成处理建议。
- 提交人工审批。
- 记录引用和 trace。

## 6. 第一个作业：Agent 场景分析表

这一讲不写代码。你的任务是学会分析一个 Agent 场景。

请用下面模板分析 “企业内部工单与知识库 Agent”。

| 项目 | 内容 |
|---|---|
| 目标用户 | 谁会使用这个 Agent？ |
| 主要入口 | 用户从哪里发起请求？飞书群聊、机器人、服务台、网页？ |
| 输入类型 | 文本、截图、文档、表格、语音？ |
| 需要的数据 | 知识库、历史工单、客户、订单、合同、权限？ |
| 需要的工具 | 查文档、查工单、查客户、创建工单、发通知、提交审批？ |
| 状态 | 任务是否需要多轮？是否需要等待审批？ |
| 输出 | 答案、建议、工单、审批单、风险报告？ |
| 风险动作 | 退款、改权限、删数据、发外部消息、修改订单？ |
| 人工介入点 | 哪些情况必须转人？ |
| 测试方式 | 如何证明它做对了？ |
| 上线方式 | shadow、copilot、semi-auto，还是 limited autonomous？ |

## 7. 一个参考答案

| 项目 | 内容 |
|---|---|
| 目标用户 | 客服、一线销售、运营、内部员工 |
| 主要入口 | 飞书机器人、服务台、群聊提及 |
| 输入类型 | 文本为主，后续可支持截图和文档 |
| 需要的数据 | 知识库、退款政策、客户资料、订单、历史工单 |
| 需要的工具 | 知识库检索、客户查询、订单查询、工单创建、审批提交 |
| 状态 | 需要记录问题分类、检索结果、审批状态、处理结果 |
| 输出 | 处理建议、引用依据、风险等级、下一步动作 |
| 风险动作 | 退款、改合同状态、修改客户权限、对外承诺 |
| 人工介入点 | 高金额退款、政策冲突、证据不足、客户投诉升级 |
| 测试方式 | 正常样例、边界样例、越权样例、工具失败样例、trace grading |
| 上线方式 | 先 shadow，再 copilot，最后只对低风险问题 semi-auto |

## 8. 这一讲的关键结论

- Agent 不是会聊天的模型，而是能连接工具、数据、状态和业务流程的软件系统。
- 企业落地优先 workflow，不要一开始追求 fully autonomous。
- 工单与知识库 Agent 是最适合作为入门主线的企业场景。
- 飞书可以作为中文企业环境的很好参照：群聊是入口，知识库是数据源，多维表格是轻量业务数据，审批是 human-in-the-loop，服务台是工单流。
- 代码库维护 Agent 很适合作为进阶作品集方向，但不适合作为第一版课程主线。
- 第一讲的核心能力不是写代码，而是判断：这个场景是否值得做 Agent，以及它的工具、状态、风险和测试边界是什么。
