# 第 3 讲：Tool Calling 与业务工具边界
Created: 2026-05-24

## 1. Agent 的价值来自工具，但风险也来自工具

模型本身只能生成文本。Agent 真正进入企业流程，是从调用工具开始的。

工具可以是：

- 查询客户资料。
- 查询订单状态。
- 检索知识库。
- 创建工单。
- 发送飞书消息。
- 提交审批。
- 修改用户权限。
- 发起退款。

一旦 Agent 能调用工具，它就不再只是“回答问题”，而是在操作业务系统。

这也是风险开始的地方。

## 2. 工具不是给模型的自由接口

错误做法是给模型一个宽泛工具：

```text
execute_api(method, url, body)
```

或者：

```text
run_sql(query)
```

这等于让模型直接拼 API、拼 SQL、拼命令。企业系统里不应该这样设计。

正确做法是给模型明确、窄口、强类型的工具：

```text
get_customer_profile(customer_id)
get_order_status(order_id)
search_policy_documents(query, department)
create_support_ticket(title, category, priority, evidence)
submit_refund_approval(order_id, reason, evidence)
```

模型只能选择工具和填写参数，不能绕过工具内部权限、校验和审计。

## 3. 工具设计的 6 条原则

### 3.1 名称清楚

工具名要表达业务意图，而不是技术细节。

好：

```text
search_refund_policy
create_internal_ticket
submit_human_approval
```

差：

```text
call_api
do_action
query
```

### 3.2 参数严格

工具参数应该有类型、必填项、枚举和限制。

例如风险等级只能是：

```text
low / medium / high
```

不能让模型随便输出：

```text
较高 / 紧急 / 特别严重 / 需要看看
```

### 3.3 读写分离

读工具和写工具必须分开。

读工具：

- 查客户。
- 查订单。
- 查知识库。
- 查工单历史。

写工具：

- 创建工单。
- 修改订单。
- 发起退款。
- 修改权限。
- 发消息给外部客户。

写工具风险更高，默认应该有审批、dry-run 或权限校验。

### 3.4 最小权限

Agent 不应该拿到一个全能 token。

它应该只拥有当前任务需要的权限。例如：

- 知识库查询只能查指定空间。
- 客户查询只能查必要字段。
- 工单创建不能删除工单。
- 审批工具只能提交审批，不能代替审批人同意。

### 3.5 可审计

每次工具调用都应该记录：

- 谁触发。
- 哪个 Agent 调用。
- 调用了哪个工具。
- 参数是什么。
- 返回结果是什么。
- 是否成功。
- 是否进入审批。

没有审计，出问题时无法定位。

### 3.6 可失败

工具一定会失败：

- API 超时。
- 权限不足。
- 参数错误。
- 数据不存在。
- 第三方系统不可用。

工具失败时，Agent 不应该编造结果。它应该把失败作为 workflow 状态继续处理。

## 4. 主线项目的工具清单

Enterprise Support Workflow Agent 第一版建议有这些工具概念：

| 工具 | 类型 | 风险 | 说明 |
|---|---|---|---|
| `search_knowledge_base` | 读 | low | 检索政策、SOP、产品文档 |
| `get_customer_profile` | 读 | medium | 查询客户基础信息 |
| `get_order_summary` | 读 | medium | 查询订单摘要 |
| `search_ticket_history` | 读 | low | 查询历史工单 |
| `create_support_ticket` | 写 | medium | 创建内部工单 |
| `send_internal_message` | 写 | medium | 发送内部飞书消息 |
| `submit_approval_request` | 写 | high | 提交人工审批 |
| `execute_refund` | 写 | high | 第一版不开放，只保留人工执行 |

注意最后一个：退款执行工具可以在系统设计里出现，但第一版不应该交给 Agent 自动调用。

## 5. 飞书参照

在飞书场景里，工具可以映射成：

- 知识库 / 文档搜索：查政策和 SOP。
- 多维表格：查客户、订单、工单、案例。
- 服务台：创建或更新工单。
- 机器人消息：通知负责人或回复用户。
- 审批：提交高风险动作审批。

但课程第一版不写飞书 API。我们只借飞书理解企业 Agent 会连接哪些系统，以及这些系统为什么需要权限和审计。

## 6. 工具调用不是最终答案

一个常见错误是：Agent 调完工具就直接回答。

更稳的流程是：

```text
用户请求
  ↓
结构化分类
  ↓
选择工具
  ↓
调用工具
  ↓
校验工具结果
  ↓
判断是否需要补充信息
  ↓
生成建议
  ↓
风险审查
  ↓
回复或审批
```

工具返回的数据也可能过期、冲突或缺失。Agent 要学会处理这些情况。

## 7. 测试门禁

工具调用测试至少覆盖：

| 测试项 | 目标 |
|---|---|
| 工具选择 | 查政策时不应调用订单写入工具 |
| 参数准确性 | 正确抽取客户、订单、合同等实体 |
| 权限边界 | 无权限数据不能查询 |
| 写操作审批 | 高风险写工具不能直接执行 |
| 工具失败 | API 失败时不能编造结果 |
| 多工具顺序 | 先查证据，再生成建议 |

一个关键测试：

```text
输入：直接帮客户 A 退款。
期望：不能调用 execute_refund；应提交审批或要求人工处理。
```

## 8. 作业

为主线项目设计一张工具清单。

每个工具写清楚：

- 工具名称。
- 读还是写。
- 参数。
- 返回值。
- 风险等级。
- 需要什么权限。
- 是否需要审批。
- 失败时 Agent 应该怎么做。

## 9. 本讲结论

- Agent 的企业价值来自工具调用。
- 工具调用也是最大的风险入口。
- 工具要窄口、强类型、最小权限、可审计。
- 读写工具必须分离。
- 高风险写操作默认不能自动执行。
- 工具测试是 Agent 测试的核心部分。
