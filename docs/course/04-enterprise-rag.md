# 第 4 讲：企业知识库 RAG
Created: 2026-05-24

## 1. 企业知识库 Agent 不是普通问答机器人

很多人把 RAG 理解成：

```text
文档切块 → 向量检索 → 塞给模型 → 回答
```

这个流程可以做 demo，但离企业落地还差很远。

企业知识库 Agent 面临的问题更复杂：

- 文档版本会变。
- 文档权限不同。
- 不同文档可能互相冲突。
- 用户问题可能需要多个来源共同回答。
- 有些问题知识库里没有答案。
- 答案必须引用来源，方便人检查。

所以企业 RAG 的目标不是“回答得像真的”，而是：

**让 Agent 基于可追溯证据回答，并在证据不足时拒答或升级。**

## 2. RAG 在主线项目里的位置

Enterprise Support Workflow Agent 需要知识库回答这些问题：

- 退款政策是什么？
- 权限申请流程是什么？
- 某类客户是否有特殊合同条款？
- 服务故障应该按哪个 SOP 处理？
- 客服遇到投诉升级时该怎么做？

这些答案不能靠模型记忆。必须从企业知识库、历史工单、合同记录或产品文档中检索。

基本流程：

```text
用户问题
  ↓
结构化分类
  ↓
生成检索查询
  ↓
权限过滤
  ↓
召回候选文档
  ↓
重排与去重
  ↓
选择证据
  ↓
基于证据生成答案
  ↓
输出引用与不确定性
```

## 3. 企业 RAG 的 6 个关键设计

### 3.1 文档来源

需要记录每个文档来自哪里：

- 飞书知识库。
- 飞书云文档。
- 服务台 FAQ。
- 历史工单。
- 产品手册。
- 合同模板。

没有来源，答案就无法追溯。

### 3.2 文档版本

政策类文档必须有版本和更新时间。

例如：

```text
refund_policy_v2.md: 2025-11-01
refund_policy_v3.md: 2026-03-15
```

如果 Agent 引用了旧政策，就可能造成业务事故。

### 3.3 权限过滤

不是所有员工都能看所有文档。

RAG 不能先检索全部文档再让模型自己决定能不能看。权限过滤应该发生在检索前或检索时。

### 3.4 引用片段

答案必须带引用。

好的引用不是“参考了知识库”，而是：

- 文档名。
- 章节。
- 更新时间。
- 相关片段。

### 3.5 冲突检测

企业文档经常冲突。

例如销售手册写“重要客户可 30 天退款”，客服 SOP 写“默认 7 天退款”。Agent 不应该强行合并成一个答案，而应该指出冲突并升级。

### 3.6 拒答能力

没有证据时要拒答。

企业 Agent 最危险的行为之一，是在证据不足时生成看起来合理的答案。

## 4. 飞书参照

飞书可以作为企业 RAG 的典型环境：

- 知识库：制度、SOP、产品文档。
- 云文档：项目方案、会议纪要、政策说明。
- 多维表格：结构化业务记录。
- 服务台知识库：FAQ 和标准处理流程。
- 群聊消息：可能包含上下文，但权限和噪音更复杂。

第一版课程不写飞书 API，但会把这些能力映射到 RAG 设计：

| 飞书能力 | RAG 设计问题 |
|---|---|
| 知识库 | 文档来源、层级、权限 |
| 云文档 | 版本、更新时间、引用 |
| 多维表格 | 结构化检索 |
| 服务台 FAQ | 标准答案和处理流程 |
| 群聊 | 非正式知识，慎重使用 |

## 5. RAG 输出格式

主线项目里的 RAG 回答建议包含：

```text
answer: 面向用户的回答
evidence: 支撑答案的引用
conflicts: 发现的冲突
unknowns: 仍不确定的信息
risk_level: 基于证据的风险等级
recommended_next_step: 下一步建议
```

这样后续 workflow 可以判断：

- 是否能直接回复。
- 是否需要查业务系统。
- 是否需要人工审批。
- 是否需要追问用户。

## 6. 测试门禁

企业 RAG 至少测试：

| 测试项 | 目标 |
|---|---|
| 召回率 | 正确文档能被召回 |
| 精确率 | 召回内容不要大量无关 |
| 引用准确性 | 答案引用的片段真实支持结论 |
| 拒答 | 无资料时不编造 |
| 版本正确性 | 优先引用最新政策 |
| 权限 | 无权限文档不能出现在上下文 |
| 冲突检测 | 冲突政策能被指出 |

示例：

```text
问题：客户是否可以 30 天退款？
期望：引用最新版退款政策；如果存在客户合同例外，提示需要查合同。
失败：只根据模型常识回答“可以”。
```

## 7. 作业

为主线项目设计一份知识库结构。

至少包含：

- 5 类文档。
- 每类文档的权限。
- 每类文档的更新时间要求。
- 引用格式。
- 无答案时的拒答规则。
- 冲突文档的处理规则。

## 8. 本讲结论

- 企业 RAG 的核心是证据，不是流畅回答。
- 权限、版本、引用、拒答和冲突检测是企业 RAG 的关键。
- 飞书知识库、文档、多维表格、服务台都可以映射到 RAG 数据源。
- RAG 输出应该服务 workflow，而不只是服务最终回答。
