# 第 5 讲：Research Agent：从问答到可验证报告
Created: 2026-05-24

## 1. 问答和研究不是一回事

普通问答追求快速回答。

Research Agent 追求的是：

- 多来源检索。
- 证据对齐。
- 冲突识别。
- 不确定性表达。
- 可供人决策的结论。

在企业场景里，很多问题不能只返回一句话。

例如：

> 客户 A 要求按合同走 30 天退款，但客服 SOP 写 7 天，我们应该怎么处理？

这个问题需要研究，而不是直接回答。

## 2. Research Agent 的输出应该长什么样

一个可验证报告至少包含：

- 问题重述。
- 已检索来源。
- 关键事实。
- 证据引用。
- 冲突点。
- 未知信息。
- 风险判断。
- 建议下一步。

推荐结构：

```text
1. 结论摘要
2. 关键证据
3. 冲突或不确定性
4. 风险等级
5. 建议动作
6. 需要人工确认的问题
```

这比一句“建议按合同处理”更适合企业。

## 3. 主线项目中的 Research Agent

在 Enterprise Support Workflow Agent 中，Research Agent 位于 RAG 和业务决策之间。

流程：

```text
结构化请求
  ↓
检索知识库
  ↓
查询历史工单
  ↓
查询客户/订单摘要
  ↓
整理证据
  ↓
发现冲突
  ↓
输出研究报告
  ↓
交给风险判断与审批节点
```

它不直接执行业务动作，只提供可审查的依据。

## 4. 区分事实、推断和建议

Research Agent 最重要的能力之一，是不要把不同性质的信息混在一起。

| 类型 | 例子 |
|---|---|
| 事实 | 最新退款政策写明默认 7 天内可退款 |
| 证据 | `refund_policy_v3` 第 2 节 |
| 推断 | 客户 A 可能属于合同例外客户 |
| 未知 | 还没有查到客户 A 的合同条款 |
| 建议 | 先查合同，再提交人工审批 |

很多 Agent 幻觉来自把推断当事实。

## 5. 飞书参照

飞书环境中的研究来源可能包括：

- 知识库政策文档。
- 云文档里的合同模板。
- 多维表格里的客户记录。
- 服务台历史工单。
- 群聊里的上下文。
- 审批记录里的历史决策。

其中群聊和历史审批要谨慎使用。它们可能是非正式信息，不一定代表最新制度。

Research Agent 应该给来源分级：

| 来源 | 可信度 |
|---|---|
| 正式制度文档 | 高 |
| 审批记录 | 中高 |
| 历史工单 | 中 |
| 群聊消息 | 低到中 |
| 用户口述 | 低，需要验证 |

## 6. 冲突怎么处理

企业里冲突很常见：

- 新旧政策冲突。
- 销售承诺和客服 SOP 冲突。
- 合同条款和标准政策冲突。
- 群聊里的经验和正式文档冲突。

Research Agent 不应该消灭冲突，而应该暴露冲突。

好的输出：

```text
发现冲突：
- 最新退款政策默认 7 天内可退款。
- 客户 A 的销售备注提到 30 天退款承诺，但未找到合同原文。

建议：
- 先查询客户合同原文。
- 若合同确认 30 天条款，提交人工审批。
```

差的输出：

```text
客户应该可以 30 天退款。
```

## 7. 测试门禁

Research Agent 要测：

| 测试项 | 目标 |
|---|---|
| 来源覆盖 | 是否检索了必要来源 |
| 引用有效 | 结论是否有证据支持 |
| 冲突识别 | 冲突文档是否被指出 |
| 未知表达 | 缺少信息时是否说明 |
| 建议合理 | 建议是否符合风险等级 |
| 不越权 | 不基于无权限资料给结论 |

评测时不要只看最终结论，还要看报告结构是否完整。

## 8. 作业

围绕这个问题写一份 Research Agent 报告模板：

```text
客户要求按 30 天退款处理，但知识库写的是 7 天。客服应该怎么处理？
```

要求包含：

- 结论摘要。
- 至少 3 类证据来源。
- 冲突点。
- 未知信息。
- 风险等级。
- 建议下一步。
- 哪些情况必须人工审批。

## 9. 本讲结论

- Research Agent 不是问答机器人。
- 企业研究输出必须区分事实、证据、推断、未知和建议。
- 冲突不是错误，隐藏冲突才是错误。
- 可验证报告是后续审批和决策的基础。
