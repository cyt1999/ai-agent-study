# 第 6 讲：Workflow 优先，而不是盲目 Autonomous
Created: 2026-05-24

## 1. 为什么企业 Agent 要先从 Workflow 开始

很多人想做 Agent，是因为想让模型“自己想办法完成任务”。这听起来强大，但企业落地时风险很高。

企业系统更需要：

- 流程可控。
- 状态明确。
- 失败可恢复。
- 高风险动作可审批。
- 每一步可审计。
- 改动后可回归测试。

这就是为什么课程主线强调：

**先 workflow，后 autonomous。**

## 2. Workflow 和 Autonomous 的差别

Workflow 是预定义路径：

```text
分类 → 检索 → 查询 → 生成建议 → 风险判断 → 审批或回复
```

模型可以参与每个节点，但整体路径由系统控制。

Autonomous Agent 是模型动态决定路径：

```text
模型自己判断下一步做什么、调什么工具、什么时候停止
```

它更灵活，但也更难测试、更难审计、更容易走偏。

## 3. 企业 Workflow 的基本结构

Enterprise Support Workflow Agent 可以设计成：

```text
入口
  ↓
请求标准化
  ↓
意图分类
  ↓
风险初判
  ↓
知识库检索
  ↓
业务数据查询
  ↓
研究报告生成
  ↓
风险复核
  ↓
动作决策
  ↓
回复 / 创建工单 / 提交审批 / 转人工
```

这里每一步都应该有明确输入和输出。

## 4. 哪些地方适合让模型判断

模型适合：

- 理解用户自然语言。
- 生成检索 query。
- 总结多来源信息。
- 判断信息是否足够。
- 生成给人的解释。
- 根据规则做初步分类。

模型不适合：

- 直接决定退款。
- 直接修改权限。
- 直接删除数据。
- 直接绕过审批。
- 直接拼 SQL 或 shell。

企业 Agent 的设计原则是：

**让模型做理解和建议，让程序控制流程和边界。**

## 5. 状态为什么重要

工单问题往往不是一轮完成。

例如：

```text
用户提出退款问题
  ↓
Agent 发现缺少订单号
  ↓
用户补充订单号
  ↓
Agent 查询订单
  ↓
发现需要合同
  ↓
等待销售上传合同
  ↓
提交审批
  ↓
审批通过后处理
```

如果没有状态，Agent 每次都要重新理解上下文。

状态至少包括：

- 请求 ID。
- 用户身份。
- 当前步骤。
- 已收集信息。
- 已调用工具。
- 检索证据。
- 风险等级。
- 审批状态。
- 最终处理结果。

## 6. 飞书参照

飞书场景下，workflow 可以映射到：

- 群聊消息触发入口。
- 机器人追问缺失信息。
- 知识库检索政策。
- 多维表格记录工单状态。
- 审批流程处理高风险动作。
- 服务台承载正式工单。
- 任务提醒负责人跟进。

这些能力组合起来，比一个纯聊天机器人更接近企业真实流程。

## 7. 常见 Workflow 模式

### 7.1 Routing

根据请求类型走不同路径。

例如：

- 产品咨询 → 知识库回答。
- 退款争议 → 查政策 + 查订单 + 审批。
- 权限申请 → 查权限规则 + 提交审批。
- 系统故障 → 创建技术工单。

### 7.2 Parallelization

并行查多个来源：

- 知识库。
- 历史工单。
- 客户信息。
- 订单记录。

再汇总结果。

### 7.3 Evaluator-Optimizer

先生成草稿，再让评估节点检查：

- 是否有引用。
- 是否有越权内容。
- 是否遗漏风险。
- 是否需要人工审批。

### 7.4 Human-in-the-loop

当风险高或证据不足时，让人介入。

这不是失败，而是企业 Agent 的正常设计。

## 8. 测试门禁

Workflow 测试要看路径是否正确：

| 测试项 | 目标 |
|---|---|
| 路由准确性 | 不同请求进入正确流程 |
| 状态转换 | 每一步状态可预测 |
| 审批路径 | 高风险请求必须进审批 |
| 失败恢复 | 工具失败后不丢任务 |
| 追问逻辑 | 信息不足时追问而不是编造 |
| 终止条件 | Agent 知道什么时候结束 |

不要只看最终回答。路径错误但答案碰巧正确，也应该判失败。

## 9. 作业

画出主线项目的 workflow。

至少包含：

- 入口节点。
- 分类节点。
- 检索节点。
- 工具查询节点。
- 研究报告节点。
- 风险判断节点。
- 审批节点。
- 回复或创建工单节点。
- 失败处理路径。

并标注每个节点：

- 输入。
- 输出。
- 是否调用模型。
- 是否调用工具。
- 是否需要测试。

## 10. 本讲结论

- 企业 Agent 应该先 workflow，再 autonomous。
- 模型负责理解和建议，程序负责流程和边界。
- 状态、审批、失败恢复是企业 Agent 的基础能力。
- Workflow 的测试要看路径，不只看最终答案。
