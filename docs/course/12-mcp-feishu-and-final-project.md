# 第 12 讲：MCP、飞书集成思路与最终项目设计
Created: 2026-05-24

## 1. 为什么最后讲 MCP

MCP 不是 Agent 框架。

它解决的是另一个问题：

**Agent 如何以标准方式连接外部工具、数据源和上下文。**

一个企业 Agent 可能要连接：

- 知识库。
- 工单系统。
- 数据库。
- CRM。
- IM。
- 审批系统。
- 代码库。
- 文件系统。

如果每个系统都写一套私有连接方式，维护成本很高。MCP 的价值在于提供统一协议，让工具、资源和 prompt 能被 Agent 客户端发现和调用。

## 2. MCP 的基本概念

MCP 里常见的能力包括：

- Tools：可执行函数，例如查询工单、查订单。
- Resources：上下文资源，例如文档、数据库记录。
- Prompts：可复用提示模板。

对企业 Agent 来说，MCP 可以作为工具连接层。

但 MCP 也带来安全风险：

- 工具污染。
- 权限扩大。
- token 暴露。
- 命令注入。
- 缺少审计。
- shadow MCP server。

所以 MCP 要后学。先学会工具边界、安全和 eval，再接 MCP。

## 3. 飞书在最终项目中的角色

我们第一版不写飞书 API 代码，但用飞书作为企业场景参照。

最终项目可以这样映射：

| 飞书能力 | 项目里的角色 |
|---|---|
| 群聊 / 单聊 | 用户入口 |
| 机器人 | Agent 入口和通知 |
| 知识库 / 文档 | RAG 数据源 |
| 多维表格 | 客户、订单、工单、历史案例 |
| 审批 | 高风险动作 human-in-the-loop |
| 服务台 | 正式工单系统 |
| 任务 | 后续跟进 |

理论课程阶段，你只需要理解这些系统如何参与 Agent 架构。

## 4. 最终项目：Enterprise Support Workflow Agent

项目目标：

构建一个企业内部工单与知识库 Agent 的完整方案。

它要能处理这类问题：

```text
客户 A 说合同里承诺可以 30 天退款，但知识库写的是 7 天。这个单子应该怎么处理？
```

它不应该直接退款。

它应该：

- 识别问题类型。
- 判断风险等级。
- 检索退款政策。
- 查询客户和订单摘要。
- 发现合同与政策可能冲突。
- 输出研究报告。
- 提交人工审批或建议转人工。
- 记录 trace 和证据。

## 5. 系统模块

最终方案包含：

### 5.1 入口层

来源：

- 飞书机器人。
- 服务台工单。
- Web 表单。

职责：

- 接收请求。
- 记录用户身份。
- 标准化输入。

### 5.2 Triage 层

职责：

- 意图分类。
- 风险初判。
- 缺失信息识别。
- 路由下一步。

### 5.3 Knowledge 层

职责：

- 检索知识库。
- 返回引用。
- 检查文档版本。
- 发现冲突。

### 5.4 Business Context 层

职责：

- 查询客户摘要。
- 查询订单摘要。
- 查询历史工单。
- 不暴露不必要敏感字段。

### 5.5 Research 层

职责：

- 整理证据。
- 区分事实、推断、未知。
- 生成可审查报告。

### 5.6 Workflow 层

职责：

- 控制流程。
- 管理状态。
- 处理失败。
- 等待审批。
- 记录最终结果。

### 5.7 Safety & Governance 层

职责：

- 权限校验。
- 高风险动作审批。
- 安全测试。
- 审计日志。
- 上线门禁。

## 6. 最终交付物

理论课程第一版的最终交付不是代码，而是一套完整项目设计文档：

- 业务场景说明。
- Agent 场景分析表。
- 结构化输出 schema。
- 工具清单和权限矩阵。
- 知识库结构和引用规则。
- Research Agent 报告模板。
- Workflow 流程图。
- LangGraph 状态和节点设计。
- Eval plan。
- Observability plan。
- Security checklist。
- Production readiness checklist。

这些文档足以指导后续项目实践。

## 7. 进阶作品集：Repo Maintenance Agent

课程最后可以把主线能力迁移到代码库维护 Agent。

映射关系：

| Support Agent | Repo Agent |
|---|---|
| 工单请求 | GitHub issue |
| 知识库 | 代码库和文档 |
| 客户/订单工具 | 文件搜索、测试运行 |
| 审批 | PR review |
| 退款风险 | 代码变更风险 |
| 服务台工单 | Pull request |
| 处理建议 | Patch plan |

这个方向非常适合工程师求职展示，但建议在主线项目之后做。

## 8. 最终项目评分标准

一个合格作品集方案应该能说明：

- 为什么这个场景适合 Agent。
- 哪些部分是 workflow，哪些部分使用模型判断。
- 工具有哪些，权限边界是什么。
- RAG 如何保证引用和版本。
- 高风险动作如何审批。
- Eval dataset 如何设计。
- Trace 记录什么。
- 如何灰度上线。
- 哪些能力第一版不做自动化。

如果这些都讲清楚，即使暂时没有代码，也已经具备企业 Agent 设计能力。

## 10. 本讲结论

- MCP 是工具连接协议，不是 Agent 框架。
- 飞书适合作为中文企业 Agent 场景参照。
- 最终项目主线是企业内部工单与知识库 Agent。
- 代码库维护 Agent 是进阶作品集方向。
- 理论课程的最终产出是一套能指导实战的企业 Agent 项目设计文档。
