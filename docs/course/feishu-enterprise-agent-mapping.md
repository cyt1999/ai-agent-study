# 飞书企业 Agent 场景映射
Created: 2026-05-24

## 1. 为什么用飞书做场景参照

很多中文企业的协作、文档、审批和工单流程都集中在飞书、企微、钉钉、语雀等平台里。

本课程选择飞书作为主要参照，不是因为课程要写飞书 API，而是因为飞书能很好地代表企业 Agent 会连接的系统类型：

- IM 入口。
- 机器人。
- 知识库。
- 文档。
- 表格。
- 审批。
- 服务台。
- 任务和通知。

这些能力组合起来，正好构成 Enterprise Support Workflow Agent 的业务环境。

## 2. 总体映射

```text
飞书群聊 / 单聊 / 服务台
  ↓
Agent 入口层
  ↓
结构化分类与风险判断
  ↓
飞书知识库 / 云文档 / 多维表格 / 服务台历史
  ↓
RAG 与业务上下文查询
  ↓
Research Agent 报告
  ↓
LangGraph Workflow
  ↓
飞书审批 / 服务台 / 机器人通知
  ↓
人工确认或低风险自动处理
```

## 3. 群聊 / 单聊

Agent 架构角色：

- 用户入口。
- 上下文来源。
- 通知渠道。

典型场景：

- 员工在群里 @机器人：“客户 A 这个退款怎么处理？”
- 客服私聊机器人：“帮我查一下这个工单进度。”
- 运营在群里发截图，请 Agent 判断应该转给谁。

设计注意：

- 群聊内容噪音高，不应直接当作正式事实。
- 群聊成员权限不同，不能随意泄露敏感信息。
- Agent 回复要控制范围，避免把敏感内容发到大群。
- 群聊适合入口和通知，不适合作为唯一事实来源。

风险边界：

- 不在群里输出客户敏感信息。
- 不根据群聊口头承诺直接执行业务动作。
- 高风险结论应转服务台或审批。

## 4. 机器人

Agent 架构角色：

- 对话入口。
- 通知入口。
- 交互式追问入口。

典型场景：

- 用户向机器人提交问题。
- Agent 发现缺少订单号，通过机器人追问。
- 审批提交后，机器人通知负责人。
- 工具失败时，机器人提醒人工接管。

设计注意：

- 机器人不是 Agent 本体，只是入口和交互通道。
- 机器人消息要避免过度打扰。
- 机器人不能绕过权限系统。

风险边界：

- 机器人不能代替审批人批准。
- 机器人不能向外部用户发送高风险承诺，除非人工确认。

## 5. 知识库 / 云文档

Agent 架构角色：

- RAG 数据源。
- 政策、SOP、产品文档来源。
- 引用证据来源。

典型场景：

- 检索退款政策。
- 检索权限申请流程。
- 检索产品功能说明。
- 检索故障升级 SOP。

设计注意：

- 必须记录文档来源、更新时间和版本。
- 必须做权限过滤。
- 必须输出引用。
- 旧文档和新文档冲突时，不能强行合并。

风险边界：

- 无权限文档不能进入模型上下文。
- 群聊复制的文档片段不能替代正式文档。
- 没有证据时要拒答或升级。

## 6. 多维表格

Agent 架构角色：

- 轻量业务数据库。
- 客户、订单、工单、历史案例的模拟数据源。
- 第一版项目实践时可以替代复杂后端系统。

典型表：

- 客户表。
- 订单表。
- 合同摘要表。
- 工单表。
- 历史处理案例表。
- Eval 样例表。

设计注意：

- 多维表格适合课程和原型，不代表最终生产数据库。
- 表格字段要做权限分级。
- 敏感字段默认不返回给模型。
- 写入表格也属于业务写操作，需要审计。

风险边界：

- 不允许 Agent 任意导出整张表。
- 不允许 Agent 修改关键字段，如合同状态、订单金额、客户权限。
- 写操作必须有权限和审批。

## 7. 审批

Agent 架构角色：

- Human-in-the-loop。
- 高风险动作确认。
- 决策审计来源。

典型场景：

- 退款审批。
- 权限开通审批。
- 对外承诺审批。
- 合同例外处理审批。

审批单应包含：

- 原始请求。
- Agent 分类结果。
- 证据引用。
- 工具调用摘要。
- 风险原因。
- 建议动作。
- 审批选项。

设计注意：

- Agent 可以提交审批，但不能替人审批通过。
- 审批结果要写回 workflow 状态。
- 审批意见可以作为后续处理依据，但不应无条件训练成长期规则。

风险边界：

- 不允许 Agent 自动通过审批。
- 不允许审批前执行高风险动作。
- 不允许隐藏证据或工具调用记录。

## 8. 服务台

Agent 架构角色：

- 正式工单系统。
- 任务承载和流转。
- 历史处理记录。

典型场景：

- 创建客户问题工单。
- 查询历史类似案例。
- 更新处理建议。
- 记录人工最终结论。

设计注意：

- 服务台比群聊更适合作为正式问题记录。
- Agent 自动创建工单时要有明确摘要、分类、优先级、证据。
- 工单状态变化应纳入 workflow 状态。

风险边界：

- 不创建无意义重复工单。
- 不覆盖人工处理意见。
- 不把未经确认的模型结论写成事实。

## 9. 任务 / 通知

Agent 架构角色：

- 后续跟进。
- 超时提醒。
- 人工接管。

典型场景：

- 审批超过 24 小时未处理，提醒负责人。
- 工具失败后创建人工跟进任务。
- 高风险问题转给客户成功团队。

设计注意：

- 通知要有频率限制。
- 超时和重试策略要明确。
- 任务应该包含上下文链接，不要只发一句“请处理”。

风险边界：

- 不要重复轰炸负责人。
- 不要把敏感信息推送到错误的人或群。

## 10. 飞书场景下的主线项目流程

```text
1. 用户在飞书群里 @机器人
2. Agent 标准化请求
3. Agent 输出结构化 triage
4. Agent 检索飞书知识库和云文档
5. Agent 查询多维表格中的客户/订单/工单摘要
6. Agent 生成 Research Report
7. LangGraph 根据风险选择路径
8. 低风险：机器人回复或创建普通服务台工单
9. 中风险：创建工单并通知负责人
10. 高风险：提交飞书审批，等待人工决策
11. 记录 trace、工具调用、证据、成本和延迟
```

## 11. 设计原则

- 飞书是入口和业务系统参照，不是 Agent 本体。
- 群聊不是正式事实来源。
- 知识库和文档要有版本和权限。
- 多维表格适合原型，但写入仍要审计。
- 审批是高风险动作的必要节点。
- 服务台适合承载正式工单流。
- 所有工具调用都要可追踪。

## 12. 后续扩展

如果后续不使用飞书，也可以迁移到：

- 企业微信：群聊、机器人、审批、客户联系。
- 钉钉：群聊、审批、宜搭、知识库。
- 语雀：知识库和文档。
- Jira / Zendesk：工单系统。
- Notion / Confluence：知识库。
- GitHub / GitLab：代码库维护 Agent。

核心设计不变：入口、数据、工具、状态、审批、评测和审计。
