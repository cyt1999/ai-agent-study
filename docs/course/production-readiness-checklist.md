# Agent 上线门禁清单
Created: 2026-05-24

## 1. 使用方式

这份清单用于判断 Enterprise Support Workflow Agent 是否可以进入下一阶段。

核心原则：

**没有测试集、trace、安全样例、人工审批、灰度策略和回滚方案的 Agent，不能接触真实生产动作。**

## 2. 上线阶段定义

### 阶段 0：Offline Eval

Agent 只跑离线测试集。

允许：

- 使用模拟数据。
- 调用 mock 工具。
- 生成建议。
- 记录 trace。

禁止：

- 接真实用户请求。
- 调用生产写工具。
- 向真实用户发送消息。
- 修改任何业务数据。

进入下一阶段条件：

- Eval dataset 覆盖正常、边界、高风险、恶意、工具失败。
- 高风险请求不会自动执行。
- 基础 trace 完整。
- RAG 答案能引用来源。

### 阶段 1：Shadow Mode

Agent 接收真实请求副本，但不影响用户和业务系统。

允许：

- 读取脱敏或授权数据。
- 生成内部建议。
- 和人工处理结果对比。
- 记录线上 trace。

禁止：

- 自动回复用户。
- 自动创建工单。
- 自动提交审批。
- 自动执行写操作。

进入下一阶段条件：

- 与人工处理结果对比达到预设准确率。
- 没有 unsafe action attempt。
- 主要失败类型已分类。
- 成本和延迟在预算内。

### 阶段 2：Copilot Mode

Agent 给员工建议，由员工确认后执行。

允许：

- 给客服/运营生成处理建议。
- 草拟回复。
- 草拟工单。
- 草拟审批请求。

禁止：

- 未经人确认直接对外发送。
- 未经人确认执行写操作。
- 自动处理高风险动作。

进入下一阶段条件：

- 人工采纳率达到目标。
- 人工修改率可接受。
- 高风险升级准确。
- 用户反馈无重大问题。

### 阶段 3：Semi-auto Mode

低风险任务自动处理，中高风险任务审批。

允许：

- 自动回复低风险知识库问题。
- 自动创建低风险内部工单。
- 自动通知内部负责人。

禁止：

- 自动退款。
- 自动修改权限。
- 自动外发高风险承诺。
- 自动导出敏感数据。

进入下一阶段条件：

- 低风险任务长期稳定。
- 审批路径稳定。
- 回滚机制可用。
- 线上持续评测稳定。

### 阶段 4：Limited Autonomous

只允许极窄、低风险、高确定性的自动化。

允许：

- 明确、低风险、可回滚任务。
- 有速率限制和熔断。
- 有持续监控和人工接管。

禁止：

- 跨权限自动化。
- 不可逆动作。
- 高金额业务动作。
- 敏感信息导出。
- 无审批高风险写操作。

## 3. 数据与权限门禁

| 检查项 | 要求 | 通过 |
|---|---|---|
| 用户身份 | 能识别发起人和权限 |  |
| 数据权限 | 检索前或检索时做权限过滤 |  |
| 敏感字段 | 默认不返回手机号、邮箱、合同金额等敏感字段 |  |
| 文档权限 | 无权限文档不能进入模型上下文 |  |
| 工具权限 | 每个工具有独立权限 |  |
| Token 管理 | 不把 token 暴露给模型输出 |  |
| 审计 | 数据访问和工具调用可追踪 |  |

## 4. 工具门禁

| 检查项 | 要求 | 通过 |
|---|---|---|
| 读写分离 | 读工具和写工具分开 |  |
| 参数校验 | 工具参数强类型、必填校验 |  |
| 最小权限 | 工具只暴露必要能力 |  |
| 写工具审批 | 写工具默认 dry-run 或审批 |  |
| 高风险禁用 | 退款、改权限、删数据第一版不开放自动执行 |  |
| 工具失败 | 失败时不编造结果 |  |
| 调用审计 | 记录工具名、参数、返回、调用人 |  |

## 5. RAG 门禁

| 检查项 | 要求 | 通过 |
|---|---|---|
| 来源记录 | 文档有来源和更新时间 |  |
| 版本识别 | 政策类文档能识别版本 |  |
| 引用输出 | 答案必须带引用 |  |
| 无证据拒答 | 无证据时拒答或升级 |  |
| 冲突检测 | 冲突文档能指出 |  |
| 权限过滤 | 无权限文档不进入上下文 |  |
| 旧文档处理 | 旧版本不应覆盖新版本 |  |

## 6. Eval 门禁

| 检查项 | 要求 | 通过 |
|---|---|---|
| 数据集规模 | 第一版至少 100 条 eval case |  |
| 正常样例 | 覆盖常见知识库和工单问题 |  |
| 边界样例 | 覆盖信息不足、模糊输入 |  |
| 高风险样例 | 覆盖退款、权限、合同、外发 |  |
| 工具失败样例 | 覆盖超时、权限不足、无数据 |  |
| 恶意样例 | 覆盖 prompt injection、越权、数据导出 |  |
| 回归机制 | prompt/model/tool/retriever 变更后重跑 |  |
| 失败分类 | 失败能归因到具体类别 |  |

建议阻断上线的失败：

- 高风险动作未升级。
- 恶意请求被执行。
- 无证据时编造答案。
- 工具失败时编造结果。
- 无权限文档进入回答。
- 审批节点被跳过。

## 7. Observability 门禁

| 检查项 | 要求 | 通过 |
|---|---|---|
| Run ID | 每次任务有唯一 ID |  |
| 节点路径 | 能看到 workflow 走过哪些节点 |  |
| 模型调用 | 记录模型、prompt 版本、输入输出摘要 |  |
| 工具调用 | 记录工具、参数、返回、错误 |  |
| 检索记录 | 记录 query、召回文档、引用 |  |
| 审批记录 | 记录审批人、状态、意见 |  |
| 成本 | 记录 token 和成本 |  |
| 延迟 | 记录整体和节点延迟 |  |
| 失败原因 | 失败能分类 |  |

## 8. 安全门禁

| 检查项 | 要求 | 通过 |
|---|---|---|
| Prompt injection | 能拦截直接注入 |  |
| Indirect injection | 检索文档中的指令不被执行 |  |
| Tool poisoning | 不信任未知工具描述 |  |
| PII 泄露 | 不输出敏感个人信息 |  |
| 权限提升 | 用户自称身份不生效 |  |
| 工具枚举 | 不暴露完整内部工具能力 |  |
| Trace 保护 | 用户不能删除或篡改审计记录 |  |
| 外发控制 | 对外消息需要人工确认 |  |

## 9. 回滚与人工接管

上线前必须定义：

- 如何关闭 Agent。
- 如何关闭某个工具。
- 如何切回人工处理。
- 如何撤回未发送的草稿。
- 如何标记错误输出。
- 如何修复后重跑 eval。
- 出现安全事件时谁负责。

## 10. 最终判断

如果一个 Agent：

- 不能解释为什么做出某个决定；
- 不能证明自己不会越权；
- 不能在工具失败时保持安全；
- 不能在高风险场景转人工；
- 不能回放 trace；
- 不能通过回归测试；

它就不应该进入生产动作。
